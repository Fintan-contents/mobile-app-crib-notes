ディープリンクの処理

ディープリンクURLをアプリケーションで処理する方法について記載します。

アプリケーションの状態とディープリンクURLの取得​

React Native Firebase Dynamic LinksではディープリンクURLを取得するためのAPIは次の2つがあります。

  • コールドスタート、ウォームスタートではgetInitialLinkでディープリンクURLを取得します。
    • iOSでは不具合に対応するためにReact Native Linkingを利用します。
  • ホットスタートではonLinkでディープリンクURLを取得します。

どちらの場合でもディープリンクURLからコンテンツリンクURLを取得し、画面遷移を判定して、コンテンツリンクURLのパラメータを処理します。コンテンツリンクURLのパラメータはディープリンクURLの構成にあるとおり、スクリーンとパラメータを受け取ります。

Universal Linksの警告にもあるようにコンテンツリンクURLが妥当なURLかどうかを検証して、ユーザに情報を表示するなど安全なアクションにとどめてください。コンテンツリンクURLが想定されない値の場合はアプリケーションで利用しません。

コンテンツリンクURLの保持​

コンテンツリンクURLをアプリケーションが取得した後、認証が必要などの理由で、コンテンツに対応するページへ遷移できないケースがあります。このような場合は、該当ページでコンテンツリンクURLを取得するために、コンテクストに設定するか画面間で持ち回る必要があります。

コンテンツリンクURLをコンテクストに設定してしまうとクリアするタイミングなどの設計が難しく、管理が煩雑になります。そのため、コンテンツリンクURLを参照する範囲を決めて、ナビゲーションのパラメータとして画面間で持ち回るように設計します。

note

ユーザー エクスペリエンスを向上させるでも記載されているようにディープリンクから該当のページへの遷移までに多くの画面をまたがないことが好ましいです。

iOS14以降でのクリップボード利用の通知​

iOSのみですが、Firebase Dynamic Linksは、アプリがまだインストールされていないときに開いたディープリンクURLを利用するために、次のようにクリップボードを利用します。

  • ブラウザで表示したプレビューページからフォールバックURLへ遷移する際に、ディープリンクURLをクリップボードにコピー
  • インストール後に初めて起動したアプリは、クリップボードの値を読み込む

iOS14以降ではクリップボードを読み取るとOSが通知を表示する仕様となっています。そのため、初めてアプリケーションを起動したときのみ、クリップボードに値が存在していると、「<アプリ名>にSafariからペースト」のような通知が表示されます。

Firebase SDKの設定でこの機能を無効にすることもできますが、インストール後にそのまま起動してディープリンクURLを読み取る機能が利用できません。ドキュメントの警告に記載があるようにmatchTypeの値がweek程度になり、このアプリではmatchTypeがuniqueでないディープリンクURLは利用しないためです。1

note

初回起動時にディープリンクから開かれたとしても、クリップボードの値がURL形式であればクリップボードを読み取ります。 ただし、起動したときのディープリングURLが優先されるため、クリップボードから読み取ったという通知は出ますが、クリップボードの値は利用されないというケースもあります。
